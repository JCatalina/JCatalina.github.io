一、表的设计

  1. 了解需求
  2. 标识实体–>表
  3. 标识属性–>字段
  4. 标识关系。
    • 一对一：主键作为两个表的公共字段
    • 一对多：主键跟非主键建立关系
    • 多对一：非主键跟主键建立关系
    • 多对多：两个表的主键 建立对应关系 储存在另一张新表

表设计出来以后，并不是最合理的结构，需要对标进行规范化

三范式：

  • 第一范式：所有字段不可再分。(如：地址 中国北京市 address 分割为 country city 但是不是所有都要遵循的，如邮寄地址，直接一个字段就可以[反三范式]到底拆不拆分，看数据是否需要汇总，排序，仅起一个字符串的作用的话不拆分)
  • 第二范式：非键字段必须依赖于键字段。(一个表描述一件事：订单表 有订单价格，订单日期，对的；但是如果有产品编号，产品价格，则不对，因为跟订单编号没有依赖关系，应放在Products中)
  • 第三范式：消除传递依赖
    在非主键字段中，如果一个字段可以推导出另一个字段，就叫传递依赖。
    例子：有学号->姓名关系表，如果成绩表中有学号，姓名共存，就有传递依赖关系(可以相互知道)，是冗余的。

（使用3范式进行规范，减少数据的冗余）(范式越高，冗余越少，但是太高查询效率会降低，呈正态分布，3范式最高)
反三范式：
例如：高考成绩表： 语文，数学，英语，总分(冗余，就是反三范式)

是规范化与性能的选择。性能更重要。

二、慢查询的定位(优化SQL，优化哪一个-》重点就是定位慢查询)

查询出数据库的状态

show status;

show [ session | global ] status like ‘com_select/insert/update’ ; 默认session会话级别的

显示慢查询条数

show status like 'slow_queries'

显示慢查询时间

show VARIABLES like 'long_query_time'

设置慢查询时间

set long_query_time = 0.5

定位慢查询(开启慢查询日志)

一旦看起慢查询日志后，日志文件位置在Mysql数据库的配置文件中my.ini文件中找，datadir="D:/mysql/Data/"

如何开启慢查询

  1. 关闭mysql服务 net stop mysql
  2. 开启慢查询指令：cmd 进入bin目录： mysqld –safe-mode –slow-query-log
  3. set long_query_time = 0.5
  4. 到慢查询日志中查看具体的慢查询sql

explain分析sql语句

1. 如果用主键去查询，自动会使用主键索引。
2. 如果创建的是复合索引，只有左边的可以用，右边的不能用。(从左往右按顺序)
3. 模糊查询的时候，%或_写在左边没有用索引，写到右边就用索引。
4. 在条件语句中使用or，or的两边的字段必须有索引，有一个没有，索引就无法使用。
5. 如果一个字段是字符型的，必须用引号引起来。
